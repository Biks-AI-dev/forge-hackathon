# Running Biks Forge locally

How to install and start every component in this repo, in order, from a
clean checkout. See `PRD.md` for what each component does and why;
`README.md` for the one-paragraph pitch. This file is just: venv, keys,
commands, in the order that actually works.

## Repo layout

```
provisioner/            POST /forge — spec in, Daytona sandbox out (PRD §4.1)
meeting-mode/            audio + files in, ForgeSpec out, auto-forges (PRD §5)
  app/                    Python backend (FastAPI)
  frontend/               Node/TypeScript recording UI — source of truth for the UI
  static/                 generated by `frontend`'s build; what the backend actually serves
agent-template/          Dev B's sandbox agent — currently a placeholder stub
poc/                     one-off Daytona SDK smoke test, not part of the running system
test-data/               fixtures (real 366-row order Excel, recon fixtures)
```

`provisioner/` and `meeting-mode/` are separate FastAPI services with
separate venvs — they're independently deployable and only talk to each
other over HTTP (`meeting-mode` calls `provisioner`'s `/forge`).

## 1. Prerequisites

- Python 3.11+ (built and tested on 3.14)
- Node.js 18+ (built and tested on 22) — only needed if you touch
  `meeting-mode/frontend/`; the backend ships with `static/` prebuilt
- A real `DAYTONA_API_KEY` (sandbox creation genuinely hits Daytona's API,
  not mocked)
- A real LLM key + endpoint for ForgeSpec generation (see below — this
  repo currently points at Doubleword.ai, not the PRD's original Kimi)
- macOS `say`/`afconvert` are only needed if you want to generate your own
  synthetic Indonesian test audio (used during development, not required
  to run the app)

## 2. `.env`

One `.env` at the repo root, read by every component (`config.py` in each
loads it via `python-dotenv`, resolving the path relative to itself, so it
works no matter which directory you run `uvicorn` from). Never commit it —
already in `.gitignore`.

```bash
cp .env.example .env
```

Then fill in:

| Var | Required | What it is |
|---|---|---|
| `DAYTONA_API_KEY` | yes | Sandbox creation, from the Daytona dashboard/sponsor booth |
| `DAYTONA_BASE_URL` | no | Defaults to Daytona's hosted API; only set if using a different region/instance |
| `LLM_API_KEY` | yes | For ForgeSpec generation |
| `LLM_BASE_URL` | yes | `https://api.doubleword.ai/v1` in this build — PRD originally specified Kimi, but `api.kimi.ai` doesn't resolve (confirmed `NXDOMAIN`) and the sponsor's key turned out to be a Doubleword.ai key, so `meeting-mode/app/llm.py` points there instead. Swap this if you have a real Kimi/Moonshot endpoint. |
| `LLM_MODEL` | no | Defaults to `openai/gpt-oss-20b` (confirmed available on the Doubleword key by listing `/v1/models` — the model id has a slash, several doc sources online show it without one, that's wrong) |
| `NOSANA_API_KEY` / `NOSANA_WHISPER_URL` | no | Optional transcription enrichment (PRD §9, stretch-only). Empty is fine — Meeting Mode falls back to local Whisper automatically. |
| `OXYLABS_USERNAME` / `OXYLABS_PASSWORD` | no | Optional pre-meeting research (PRD §2 stretch). Empty is fine — the feature no-ops with a warning, never blocks the pipeline. |
| `OXYLABS_SCRAPER_URL` | no | Defaults to `https://realtime.oxylabs.io/v1/queries` — this is Oxylabs' own documented "Realtime" Web Scraper API endpoint (confirmed against `developers.oxylabs.io`, not sponsor-provided and not guessed). Only set this if a sponsor gives you a different endpoint. **Don't set it to an empty string in `.env`** — `KEY=` with nothing after it counts as "present" to Python's `os.environ`, which defeats the code's fallback default (this exact bug happened once, see `meeting-mode/app/config.py`'s comment on that line). Leave the line out entirely, or actually put a URL. |

Auth note: Oxylabs' real Web Scraper API uses HTTP Basic Auth
(username/password), not a bearer token — confirmed against their docs.
If a sponsor only ever hands you a single token, it won't work against
the real API as-is; `OXYLABS_USERNAME`/`PASSWORD` are what's wired up.

## 3. Provisioner

```bash
cd provisioner
python3 -m venv .venv
./.venv/bin/pip install -r requirements.txt
./.venv/bin/uvicorn app.main:app --port 8899 --log-level info
```

Verify:
```bash
curl http://127.0.0.1:8899/health
# {"status":"ok"}
```

Leave this running in its own terminal. Meeting Mode's default
`PROVISIONER_URL` is `http://127.0.0.1:8899` — if you run it on a
different port, set `PROVISIONER_URL` in `.env` to match.

**Do not restart this process while sandboxes are live.** Its
`slug → sandbox_id` registry is in-memory only (PRD §4.1's documented
scope); a restart forgets every sandbox it's tracking, so the next forge
for an already-forged business creates a duplicate instead of replacing
it, orphaning the old one.

## 4. Meeting Mode

Two parts: the Python backend (own venv, doesn't share `provisioner`'s)
and the browser UI, which is a Node/TypeScript project under
`meeting-mode/frontend/` that **builds into `meeting-mode/static/`** —
that's the directory the backend actually serves, it's generated output,
not hand-written.

### Backend

```bash
cd meeting-mode
python3 -m venv .venv
./.venv/bin/pip install -r requirements.txt
./.venv/bin/uvicorn app.main:app --port 8901 --log-level info
```

First run downloads the local Whisper `tiny` model (a few hundred MB,
cached under `~/.cache` afterward — only slow once).

Verify:
```bash
curl http://127.0.0.1:8901/health
# {"status":"ok"}
```

### Frontend (recording UI)

`meeting-mode/static/` already ships built in this repo, so the backend
works out of the box without touching Node at all. You only need this
step if you're changing `frontend/src/*.ts` — rebuild after any change,
the backend never compiles TypeScript itself:

```bash
cd meeting-mode/frontend
npm install
npm run typecheck   # tsc --noEmit, catches type errors before you ship them
npm run build        # writes straight into ../static — restart is not
                      # needed, uvicorn serves whatever's on disk per request
```

For active frontend development, `npm run dev` runs a Vite dev server on
`:5173` with hot reload, proxying `/jobs`, `/research`, `/health` to the
real backend on `:8901` (see `frontend/vite.config.ts`) — faster than
rebuilding on every change, but the backend must already be running.

Open `http://127.0.0.1:8901/` in a browser for the actual UI: pre-meeting
research (optional), record-or-upload audio, attach PDF/Excel client
files. **Stopping the recording auto-starts the forge** the moment both
the audio and at least one client file are present — no separate confirm
click, matching the "no manual ForgeSpec confirmation" rule further up
the pipeline. The Forge button stays visible as a manual fallback (e.g.
after Discard & re-record).

Both services need to be running for a real forge to complete — Meeting
Mode's pipeline calls the Provisioner's `/forge` as its last step.

With both services up, forge the PRD's sample recon spec:

  "workflow": "recon",
  "persona": {"agent_name": "Nusa", "language": "id", "tone": "tenang, jelas", "owner_name": "Pak Dharma", "admin_name": "Mbak Sari"},
  "business": {"name": "Dapoer Nusantara", "outlets": ["DN1", "DN2"], "bank": "BCA"},
  "channels": [
    {"name": "CASH", "hits_bank": false},
    {"name": "QRIS", "fee_rate": 0.007, "settle_days": 1, "assumed": true},

For the full pipeline (audio → transcription → LLM → forge), use the
browser UI at `http://127.0.0.1:8901/` — that exercises everything.

## 7. Parallel sandbox verification (stretch, optional)

Only meaningful once the Provisioner is confirmed stable on its own:

```bash
cd provisioner
./.venv/bin/python3 parallel_verify.py
```

Forges three deliberately distinct businesses concurrently, confirms
distinct `sandbox_id`/`chat_url`/`slug` for each, health-checks all three
at once, and checks for cross-sandbox data leakage by pulling each
sandbox's actual `spec.json`. Cleans up after itself unless you pass
`--keep`.

## 8. Running on a server instead of your laptop

The team's own plan (`ATTACK_PLAN.md`) is laptop-only: Daytona is already
cloud, sandboxes get their own public preview URLs, and only the person
running Meeting Mode needs to reach `provisioner`/`meeting-mode` locally.
**You don't need a server for the demo itself.** Use this section only if
you specifically want a shared instance — e.g. so a remote teammate can
run Meeting Mode without you screen-sharing your laptop.

**This must be a different machine from the production VPS.**
`deploy/standalone-bot/` runs there as a systemd service — rule #1 from
the start of this build is don't touch it, and that includes not
co-locating this repo's services on the same box out of convenience. Spin
up a separate small VM (any provider) or use a teammate's own laptop.

### Clone

```bash
git clone <this-repo-url>
cd forge-hackathon
cp .env.example .env
# fill in .env — see the table in section 2 above; scp it over or paste
# manually, don't commit it, don't send it through Slack/email in plaintext
```

### Ports

| Service | Port | Needs to be reachable by |
|---|---|---|
| Provisioner | `8899` | Only `meeting-mode` (same host, `127.0.0.1` is enough — never needs to be public) |
| Meeting Mode | `8901` | Whoever is running the meeting (the browser hitting the UI) |

Only **8901** needs to be opened in a firewall/security group if you want
someone else to reach it over the network. Keep 8899 bound to
`127.0.0.1` — the Provisioner has no auth of its own (PRD's accepted
scope: "sandbox URLs are unguessable enough for a weekend," §4.4 — that
logic doesn't extend to the Provisioner's own `/forge`, which has no such
protection, so don't expose it beyond localhost).

If you do open 8901 publicly, put it behind HTTPS (a quick Caddy or nginx
reverse proxy is enough) rather than serving plain HTTP with a mic-permission
page over the open internet — browsers also block `getUserMedia` on
non-HTTPS origins other than `localhost`, so **MediaRecorder plain won't
work over a bare public `http://` URL at all** — this is a hard browser
requirement, not a style preference. `https://localhost` and SSH port
forwarding both sidestep it without needing a cert.

### Run it persistently

The `nohup ... &` pattern used throughout this doc's testing works but
dies with the SSH session unless you also `disown` it, and doesn't survive
a reboot. For a hackathon day, a `tmux`/`screen` session per service is
simplest:

```bash
tmux new -s provisioner -d 'cd provisioner && ./.venv/bin/uvicorn app.main:app --port 8899 --log-level info'
tmux new -s meeting-mode -d 'cd meeting-mode && ./.venv/bin/uvicorn app.main:app --port 8901 --log-level info'
```

Reattach with `tmux attach -t provisioner` to watch logs; detach with
`Ctrl-b d`.

### Running it locally (recap)

If you're not deploying anywhere and just want it on your own machine —
that's sections 3–4 above, unchanged: two terminals, one venv each,
`uvicorn app.main:app --port 8899` then `--port 8901`, open
`http://127.0.0.1:8901/` in a browser. No server, no clone-elsewhere step,
no port-opening — this is the actual demo-day path per `ATTACK_PLAN.md`.

## Troubleshooting

- **"address already in use" on startup**: something's still bound to
  that port from a previous run. `lsof -ti:8899 | xargs kill -9` (swap
  the port). Background processes started with `&` in one terminal
  session don't always die when you expect — check with `lsof` rather
  than assuming.
- **Orphaned sandboxes after a crashed test run**: list and clean up
  directly —
  ```bash
  cd provisioner
  ./.venv/bin/python3 -c "
  from app.daytona_ops import make_client
  d = make_client()
  for sb in d.list():
      sb.delete(timeout=30, wait=True)
      print('deleted', sb.id)
  "
  ```
- **`api.kimi.ai` / Kimi errors**: expected — see the `LLM_BASE_URL` row
  above. This build uses Doubleword.ai, not Kimi.
- **Meeting Mode job stuck in `extracting_forgespec` forever**: this was
  a real bug (`asyncio.wait_for` racing an uncancellable thread leaked a
  bare `CancelledError`) and is fixed — every job now reaches `ready` or
  `failed` no matter what. If you see this on current code, something new
  broke; check `pipeline.py`'s top-level `except BaseException` backstop
  is still in place.
- **Local Whisper transcribes Indonesian speech oddly**: check the audio
  itself first — a garbled non-Indonesian voice reading Indonesian text
  (e.g. macOS `say` without `-v Damayanti`) produces genuinely bad input,
  not a transcription bug.
